# Call Me Maybe: Introduction to Function Calling in LLMs
![Constrained Decoding Architecture](constrained_decoding.png)

## Description

The primary goal of this project is to bridge the gap between unstructured human language and structured computer execution. When given a query like *"What is the sum of 2 and 3?"*, traditional small language models (like the 0.6B parameter model used here) often struggle to produce syntactically valid JSON consistently. 

This project solves that challenge by implementing a **Constrained Decoding** layer. Instead of hoping the model formats its response correctly, my program guides the model token-by-token. The output is guaranteed to be 100% compliant with the schemas defined in the functions definition file.

## Instructions


This project requires **Python 3.10** or later. Dependencies are managed using `uv`.

### Installation
To install the project dependencies in a virtual environment:
```bash
uv sync
```

### Execution
Run the program by specifying the functions definition file, the test prompts file, and the output path:
```bash
uv run python -m src \
  --functions_definition data/input/functions_definition.json \
  --input data/input/function_calling_tests.json \
  --output data/output/function_calls.json
```

or you can run it directly with:
```bash
make run
```
by default this will run the program with the default Qwen model 'Qwen/Qwen3-0.6B' but you can run it with the bonus model ' Qwen/Qwen2.5-1.5B-Instruct' by running:
```bash
make bonus
```

### Cleaning Up
To clean temporary files, cache directories (`__pycache__`, `.mypy_cache`), and build folders:
```bash
make clean
```

### Code Style & Verification
To run static analysis and type checks:
```bash
make lint
```

## Algorithm explanation

The core engine relies on **Constrained Decoding** via **Logits Masking** managed by a **Parser State Machine**.

```text
[ Context + Prefix ] ──> [ LLM Logits (151,643 scores) ]
                                 │
                                 ▼
                     [ Apply State-Based Mask ] (Set invalid tokens to -inf)
                                 │
                                 ▼
                     [ Select Argmax Winner ] (Integer ID)
                                 │
                                 ▼
                     [ Append ID to Tokens List ] ──> Repeat / Decode
```

1. **State Tracking:** The program monitors where it is in the JSON structure (e.g., `SELECTING_FUNCTION`, `WRITING_PARAMETER_KEY`, or `GENERATING_VALUE`).
2. **Logit Interception:** At each generation step, the model outputs raw scores (logits) for the entire vocabulary ($151,643$ tokens). 
3. **Masking:** Based on the current state, I identify all valid next token IDs (e.g., if expecting a number, only digits, dots, or minus signs are valid). The logit scores of all **invalid** tokens are set to $-\infty$.
4. **Argmax Selection:** I find the index of the highest score in my masked logits array. The model is forced to choose a valid token.
5. **JSON Skeleton Insertion:** To optimize speed and prevent structural errors, static JSON syntax (like `{`, `"name":`, `", "parameters": {`, and `}`) is appended directly by my Python code rather than being generated by the LLM.

## Design decisions

* **Dynamic Schema Parsing:** Instead of hardcoding rules for a specific set of functions, the program parses the schema properties dynamically from `functions_definition.json`. This ensures compatibility with any new function list introduced during grading.
* **Pure Python Logit Filtering:** For efficiency and to avoid heavy external dependencies, I use a selective indexing strategy: `max(valid_token_ids, key=lambda idx: logits[idx])`. This is mathematically identical to masking all other values to $-\infty$ but operates in microseconds by avoiding loops over the entire $151,643$ vocabulary array.
* **Pre-computation (Caching) at Startup:** I load and scan the `vocab.json` file once at startup to pre-filter and group token IDs (e.g. creating the list of purely numeric token IDs). During the live decoding loop, I look up this cached list instantly.
* **State Machine for JSON Delimiters:** I use JSON brackets and quote boundaries as transitions to determine when a parameter value is complete, immediately switching states or breaking the loop.

## Challenges faced

### Challenge 1: The BPE Space Character Issue
**Problem:** Direct string concatenation of vocabulary keys produced raw byte symbols like `Ġ` instead of spaces and `Äł` instead of special symbols.
**Solution:** Instead of concatenating raw vocabulary strings, when i want to append the llm_output to my main prompt i replace `Ġ` with spaces

### Challenge 2: Re-Encoding Bottleneck
**Problem:** In early tests, translating tokens to strings, appending them to the prompt, and calling `model.encode()` at every step caused massive CPU slowdowns, taking 1 minute per prompt.
**Solution:** I modified the loop to work entirely with integer token IDs (`list[int]`). I only call `model.encode()` once at startup, append the integer output IDs directly, and call `model.decode()` once at the end. This reduced execution time by over 90%.

### Challenge 3: Repetition Loops in Numbers
**Problem:** Small models sometimes get stuck generating infinite strings of the same digit (e.g. `111111...`) because the repetition bias overrides the logit score of the comma or bracket delimiter.
**Solution:** I added a length-limit safety check inside the number constraint. If a number exceeds 20 digits, the code forces a stop by setting all digit logits to $-\infty$ and allowing only the delimiter.

### Challenge 4: Empty Prompts
**Problem:** Passing an empty prompt string caused the model to hallucinate random function selections.
**Solution:** I implemented Pydantic schema validation at the input parsing stage to check for `min_length=1`. Empty prompts are blocked and raised as errors before ever reaching the LLM.

## Testing strategy

I validated the project using the following manual and automated tests:
1. **Schema compliance:** Verified that the generated JSON matches the structure of `functions_definition.json` exactly.
2. **Type validation:** Checked that string parameters are correctly quoted, and numbers do not contain letters.
3. **Empty inputs:** Passed empty strings to ensure the validation code catches them and prints clean, non-crashing error messages.
4. **Grading Suite Simulation:** Ran the full batch of 11 prompts under the 5-minute time limit to ensure the code executes cleanly and writes the result file correctly.

## Example usage


Given the prompt: *"What is the sum of 2 and 3?"* and the function definition for `add_numbers` in `functions_definition.json`, the program will generate the following output in `output_file.json`:

```json
{
  "prompt": "What is the sum of 2 and 3?",
  "name": "add_numbers",
  "parameters": {
      "a": 2,
      "b": 3
    }
}
```

my code is reuseable and can handle any function definitions and prompts as long as they are valid according to the defined schemas. The constrained decoding mechanism ensures that the output will always be syntactically correct JSON, regardless of the complexity of the functions or the prompts.

you can clone the repository and run it with your own function definitions and prompts by following the instructions in the "Instructions" section above. Just make sure to update the paths to your custom `functions_definition.json` and `function_calling_tests.json` files when running the program.
